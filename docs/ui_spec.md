# 3ペインUI仕様

対象読者: フロントエンド（`frontend/`, React + Vite + TypeScript）に触れる開発者。画面レイアウトとUX導線のみを扱う。コンポーネント構成は `docs/architecture.md` 8節参照。

## 1. 3ペイン共通レイアウト・グリッド構造

デスクトップおよびタブレット環境において、画面の横幅（デッドスペース）を情報の深掘りにフル活用するためのグリッドシステム。

* **左ペイン（幅固定：小 / `flex-basis: 240px`）:** メインメニュー（ホーム、通知、検索、プロキシ設定、投稿ボタンなど）。
* **中央ペイン（幅固定：中 / `flex-basis: 600px`）:** メインのコンテンツストリーム。
* **右ペイン（幅固定：中〜大 / `flex-basis: 500px`）:** ダイナミック・コンテキスト領域（画面のルートに応じて裏でシームレスに変形する）。

---

## 2. 各画面における右ペインの動的コンテキスト・スイッチ

ユーザーが現在見ている中央ペインのコンテキスト（画面状態）をフロントエンドの状態マネージャーが監視し、右ペインの内容を最適化する。

### 2.1 【タイムライン画面】表示時
* **右ペインの構造:** タブ切り替え式。デフォルトタブは【クイック通知】（トレンド集計はバックエンド未実装のプレースホルダのため、機能している通知を優先表示する方針）。
  * **タブ1: 【クイック通知】（デフォルト）** 自分へのリプライやリアクションの簡易ストリーム。中央のメインTLを消費しながら、自分への反応を右目でリアルタイムに監視できる。DB永続化済みで新着順の無限スクロールに対応（`docs/protocols.md` 7節）。
  * **タブ2: 【トレンド＆検索】** 検索窓 ＋ 現在の二大宇宙（AP/ATP双方）の集計から算出されたリアルタイムトレンドワードランキング（トレンド集計は未実装、検索のみ機能）。

### 2.2 【ユーザー詳細画面】表示時（左右並列プロフィールレイアウト）
Misskey等に代表される「自己紹介やメタデータが縦に非常に長いプロフィール」を開いた際、スクロール地獄を完全に破壊するための特権レイアウト。
* **中央ペイン:** ユーザーのヘッダー、アイコン、Bio、各種プロトコルアイデンティティ（DID/URI）、および「ピン留めされたポスト」を固定描画。
* **右ペイン:** **そのユーザーのポストリスト（タイムライン）を最初から全面描画。**
* **スクロール挙動:** 中央ペインと右ペインのスクロール機構を完全に独立（`overflow-y: auto`）させる。右ペインをどれだけ過去にスクロールしても、中央のプロフィール（フォローボタン等）は画面に固定され続ける。

### 2.3 【ポスト詳細（会話ツリー）画面】表示時
* **中央ペイン:** ターゲットポストを中心に、上に「リプライ元の垂直遡りツリー」、下に「直系リプライ・引用ポスト群」を表示。
* **右ペイン (タブ切り替え式):**
  * **モード1: 【投稿主の前後の投稿】** ターゲットポストの前後数件のタイムラインを表示。他人のリプライに邪魔されることなく、投稿主の「連投（ツリー投稿）」や「直前のリポスト」の文脈を単一画面で一気読みできる。
  * **モード2: 【絵文字リアクション ＆ メンツ一覧】** ポストに付いたすべての絵文字（Fediのカスタム絵文字、ATPのLike等）をグリッド集計表示。絵文字をクリックすると、それを打ったアクター（アイコン・名前）のリストが下にズラリと表示される。

### 2.4 右ペインにおけるサブタブ・ステートの維持ルール
* ユーザーが中央ペインで「ポストA」から「ポストB」へと詳細画面を切り替えた際、右ペインのコンテキスト自体はリセットされるが、**「現在アクティブなサブタブのインデックス（例: 『投稿主の前後の投稿』モード）」はセッション内で維持（リテイン）**される。
* これにより、タイムライン上のポストを上から順番にクリックしていくだけで、画面を切り替えることなく、常に右ペインで「そのポストを投げた人の前後の文脈」を連続して覗き見ることができる超高効率なブラウジングUXを実現する。

## 3. ブリッジユーザーのインターセプション（案内）導線

プロトコルの歪みによって生まれた「影武者アクター」をユーザーが閲覧・操作した際、フロントエンドが能動的に介入するUX仕様。

### 3.1 本尊へのワープ機能
* バックエンドから返ってきたアクターオブジェクトの `bridge_real_actor_id` が埋まっている（＝自分が影武者である）場合、プロフィール画面の最も目立つ位置に「**本尊（オリジナルアカウント）はこちら**」というプロトコルアイコン付きのワープボタンを強制的にレンダリングする。

### 3.2 フォロー時のインターセプト
* ユーザーが影武者アクターの「フォロー」ボタンを押した瞬間、フロントエンドは即座に配送を処理せず、以下の確認モーダル（ダイアログ）を割り込ませる。

> **「このアカウントは、Bluesky（またはFedi）のユーザーがブリッジ経由で投影されている影武者です。 seiranの機能をフルに活用するため、本尊（オリジナルアカウント）を直接フォローすることをおすすめします。本尊をフォローしに行きますか？」**
> * [ボタンA]: 本尊のプロフィールへ移動してフォロー
> * [ボタンB]: そのまま影武者をフォロー（互換維持用）

## 4. グローバル通知（トースト）

3ペインのレイアウトとは独立して、画面右下に固定表示するトースト通知（`ToastContext`）。エラー/成功/情報の3種、複数スタック可能、数秒後に自動で消える（手動での閉じるボタンもあり）。

- ネイティブの `alert()` は使わない。API操作の結果通知（リポスト失敗・削除完了等）はすべてトーストに統一する
- フォーム内のバリデーションエラー（ログイン失敗等）は、フォームに近い場所へのインライン表示のままでよく、トーストへ強制的に寄せる必要はない
- 表示文言の多言語化・エラーコードの解釈は `docs/architecture.md` の「API エラーレスポンス方針」を参照
